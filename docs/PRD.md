# 청담 PRD

## 1. 제품 개요

청담은 경북대학교 컴퓨터학부 공지사항을 근거로 답변하는 RAG 기반 공지 어시스턴트이다. 사용자는 자연어로 학사 일정, 수강신청, 졸업요건, 장학, 행정 절차를 질문하고, 청담은 최신 공지 본문과 첨부파일에서 근거를 찾아 출처와 함께 답변한다.

현재 제품의 핵심 가치는 "검색 대신 질문"이다. 단순 챗봇이 아니라, 공지 본문과 첨부파일을 크롤링, 파싱, 벡터 인덱싱, 그래프 인덱싱까지 반영한 뒤 답변해야 한다.

## 2. 목표

- 학생이 공지사항과 학사 문서를 직접 뒤지지 않고 필요한 조건과 일정을 빠르게 확인한다.
- 답변마다 출처를 노출해 사용자가 원문 공지로 검증할 수 있게 한다.
- 관리자는 웹에서 전체 동기화를 실행하고 진행 상태를 확인할 수 있다.
- 홈서버 배포 환경에서 재시작 후에도 서비스가 자동 복구된다.

## 3. 비목표

- 대학 전체 포털 로그인 연동
- 개인별 수강 가능 여부의 최종 판정
- 공지 원문을 대체하는 법적 효력 있는 안내
- 여러 학과/기관 전체 크롤링
- 모바일 앱 배포

## 4. 사용자

- 재학생: 수강신청, 졸업, 장학, 행사, 행정 일정을 빠르게 확인한다.
- 학부 조교/운영자: 최신 공지가 검색 DB에 반영됐는지 확인하고 수동 동기화를 실행한다.
- 프로젝트 평가자: RAG 답변, 출처, 그래프 근거, 배포 안정성을 검증한다.

## 5. 핵심 사용자 흐름

1. 사용자가 질문을 입력한다.
2. 백엔드는 세션을 생성하거나 기존 세션에 사용자 메시지를 저장한다.
3. 하이브리드 RAG가 Chroma 벡터 검색과 Neo4j 그래프 검색을 수행한다.
4. LLM은 검색된 문맥만 근거로 답변을 생성한다.
5. 프론트는 답변, 출처 카드, 지식 그래프를 스트리밍 형태로 표시한다.
6. 대화 기록은 SQLite에 저장되어 새로고침 후에도 조회할 수 있다.

## 6. 데이터 동기화 흐름

1. 공지 목록 크롤링: `pipeline/00_crawler.py`
2. 첨부파일 다운로드: 공지별 파일을 `data/attachments`에 저장
3. 파싱: `pipeline/01_parser.py`
   - 공지 본문
   - PDF, DOCX, HWP/HWPX, XLSX 등 첨부파일
4. 벡터 DB 빌드: `pipeline/02_vector_db.py`
   - `notice`
   - `notice_content`
5. 그래프 DB 빌드: `pipeline/03_graph_db.py`
   - Document, Chunk, Entity, 관계 생성
   - 운영 웹 동기화에서는 `--rebuild`로 실행해 오래된 테스트 문서 노드를 제거
6. RAG 캐시 갱신: 백엔드 프로세스가 새 인덱스를 읽도록 캐시 초기화

웹의 "지금 동기화" 버튼은 위 전체 흐름을 실행해야 하며, 크롤링만 실행하면 제품 요구사항을 만족하지 않는다.

## 7. 기능 요구사항

### F1. 채팅

- 사용자는 질문을 입력하고 SSE 스트리밍으로 답변을 받는다.
- 답변은 공지 기반 검색 결과가 없으면 모른다고 말해야 한다.
- 에러 발생 시 사용자에게 재시도 가능한 메시지를 보여준다.
- 같은 요청이 진행 중일 때 중복 전송을 막는다.

### F2. 출처

- 모든 답변은 가능한 경우 출처 카드를 포함한다.
- 공지 본문(`notice_content`)과 공지 첨부파일(`notice`)은 모두 `공지`로 표시한다.
- 평가/실험용 매뉴얼 문서는 운영 웹 인덱스에 포함하지 않는다.
- 공지 출처는 제목, 날짜, 원문 URL을 포함한다.
- 같은 공지가 여러 청크로 검색돼도 출처 카드는 중복 표시하지 않는다.

### F3. 하이브리드 RAG

- 기본 검색은 Chroma 벡터 검색을 사용한다.
- 그래프 검색은 벡터 검색 결과 본문에 실제 등장한 엔티티를 중심으로 보조 근거를 제공한다.
- 그래프 근거는 본문과 동떨어진 관계를 답변에 주입하지 않아야 한다.
- 검색 결과 메타데이터는 백엔드까지 보존되어야 한다.

### F4. 동기화

- 웹에서 수동 전체 동기화를 시작할 수 있다.
- 동기화 상태는 `idle`, `crawl`, `download`, `parse`, `vector`, `graph`, `reload`, `done`, `error`로 조회된다.
- 동기화 중 중복 실행은 409로 거절한다.
- 실패 시 어느 단계에서 실패했는지 상태 API가 반환한다.
- 운영 배포에서는 `SYNC_ADMIN_TOKEN`을 가진 관리자만 동기화 API를 호출할 수 있어야 한다.

### F5. 대화 기록

- 세션 목록, 세션 상세, 세션 삭제가 가능해야 한다.
- assistant 메시지는 답변, 출처, 그래프 데이터를 함께 저장한다.
- 현재 MVP는 SQLite를 사용한다. 단일 홈서버 배포에서는 허용되지만, 다중 인스턴스 운영 시 PostgreSQL로 전환한다.

### F6. 설정 화면

- 설정 화면은 실제 동작하는 기능만 보여준다.
- MVP 범위에서는 데이터 동기화 상태와 서비스 정보만 제공한다.
- 계정, 알림, 도메인 관리, 출처 정책 토글은 실제 구현 전까지 노출하지 않는다.

## 8. 비기능 요구사항

- 정확성: 답변은 검색된 근거 밖의 단정을 피한다.
- 신선도: 관리자가 동기화 후 새 공지가 검색과 답변에 반영되는지 확인할 수 있어야 한다.
- 관측성: 동기화 단계와 에러는 API로 확인 가능해야 한다.
- 보안: 운영 환경에서 `.env`와 API 키는 이미지나 Git에 포함하지 않는다.
- 성능: 일반 질문은 30초 안에 응답 시작을 목표로 한다.
- 복구성: Docker Compose 재시작 후 Neo4j, backend, frontend가 자동 복구된다.

## 9. API 계약

- `GET /health`
  - `{ "status": "ok" }`
- `POST /api/chat`
  - request: `{ "sessionId": string | null, "message": string }`
  - response: `text/event-stream`
  - event types: `session`, `text`, `sources`, `graph`, `done`, `error`
- `GET /api/history`
  - 세션 목록 반환
- `GET /api/history/{sessionId}`
  - 세션 메시지 반환
- `DELETE /api/history/{sessionId}`
  - 세션 삭제
- `GET /api/meta`
  - `{ "lastCrawledAt": string, "totalAnnouncements": number }`
- `POST /api/crawl`
  - 전체 동기화 시작
- `GET /api/crawl/status`
  - 현재 동기화 상태 반환

## 10. 데이터 모델

- `Session`
  - `id`, `title`, `created_at`, `updated_at`
- `Message`
  - `id`, `session_id`, `role`, `content`, `sources_json`, `graph_json`, `followups_json`, `is_error`, `created_at`
- Chroma metadata
  - `doc_key`, `chunk_id`, `file_name`, `source_type`, `notice_id`, `notice_num`, `notice_title`, `date`
  - `notice_id`는 게시글 URL의 `wr_id`를 사용한다. 목록 번호인 `notice_num`은 중복될 수 있어 고유키로 쓰지 않는다.
- Neo4j
  - `Document`, `Chunk`, `Entity`
  - `HAS_CHUNK`, `MENTIONS`, 도메인 관계, `CO_OCCURS`

## 12. 수용 기준

- `26학년도 여름 계절 수강신청 일정 알려줘` 같은 공지 기반 질문에 대해, 관련 공지 출처가 `공지` 카테고리로 표시된다.
- `/api/crawl` 실행 후 `crawl/status`가 `done`이 되면 새 공지 수와 검색 결과가 갱신된다.
- 출처 URL을 클릭하면 원문 공지로 이동한다.
- 설정 화면에는 실제 동작하지 않는 계정/알림 토글이 보이지 않는다.
- 백엔드와 파이프라인 파일은 `python -m py_compile`을 통과한다.
- 프론트는 `npm run typecheck`와 `npm run build`를 통과한다.
